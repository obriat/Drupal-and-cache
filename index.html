What is cache?

Store the result of slow calculations so that they do not need to be executed again and so improve performances.

Cache mecanisms upstream of Drupal

The first cache is the browser one

  • The url is the browser cache key
  • The browser will only check for updates after max-age seconds (or at expires)
  • It will get a new content or a 304 response.
  • The checksum used is usualy ETag (or Last-Modified)
  • Vary indicates alternate version of cache content (accepted encoding, ...)

Network caches

On their way to Drupal, requests could pass through:

  • Proxies: which use s-max-age headers
  • Content Delivery Network (CDN) : Akamaï, Cloudflare, ...
  • Varnish:
    • Drupal best buddy
    • Works like a proxy (default cache ttl is 2mn)
    • Swiss Army knife for HTTP headers and request
    • Could serve your website even if its down (grace mode)
    • Could replace Internal Page Cache module
  • They could use X-Drupal-Cache-Tags headers
  • They can be managed with purge module and its sub modules (varnish_purge, akamai, ...) and HTTP Cache Control module
s is for "shared" ()

Internal Page Cache

Anonymous cache

  • Manage by the Internal Page Cache module
  • It caches fully rendered HTML responses
  • Only for anonymous vistors (no session)
  • Really fast
  • Use URL as the cache key
  • Use cache tags, but not cache contexts or max-age*
  • Respect the Expires header on the response
  • As seen before, it could be disabled if an external page cache system (like Varnish) is used
  • Degugging HTTP Header: X-Drupal-Cache HIT or MISS

*https://www.drupal.org/node/2352009 + https://www.drupal.org/project/cache_control_override

Dynamic Page Caching

Dynamic Page Cache

  • Dynamic Page Cache module
  • Much slower than Anonymous cache
  • Use cache tags, cache contexts and max-age "bubbling from page contents
  • It does not cache page contents that are too dynamic, it replaces them by placeholders before storing cache
  • Just before sending cache it replaces them with their callback
  • Works for anonymous and authenticate users

Render API cache system

Render API cache system

  • The page content is build with the render API, it's where most of the cache meta-data come from.
  • Metadata are store inside the #cache key, they bubble from bottom to top arrays

Cache metadata

  • Cache Contexts: Allows cache variations according to: theme, language, user roles, permissions, URL, QS, timezone,...
  • Cache Tags: Allows to invalidate all caches tagged with them: node:x, config:, user:x, library_info, route_match, node_list, ...
  • Max Age:
    • -1: Permanent
    • X: Age in seconds
    • 0: Do not cache (once again* this will not change the http response or the anonymous cache duration.

Bubbling

  • These cache metadata aggregate to form the whole page cache metadata
  • The result is used by Dynamic Page Cache & Internal Page Cache
  • External cache like Varnish could use X-Drupal-Cache-Tags
  • HTTP header are also usefull for debuggin
  • The renderviz module is a proof of concept for displaying/debugging cache metatags

Code examples : setting cache metadata on Render Array

$config = \Drupal::config('system.site');
$current_user = \Drupal::currentUser();

$build = [
'#markup' => t('Hi, %name, welcome on @site!', [
  '%name' => $current_user->getUsername(),
  '@site' => $config->get('name'),
]),
'#cache' => [
  'contexts' => [
    'user', // Will be proccess by #lazybuilding
  ],
  'tags' => $config->getCacheTags(), // Same as 'tags' => ['config:system.site']
],
];

// Another way to add cache depency.
$renderer = \Drupal::service('renderer');
$renderer->addCacheableDependency(
$build,
\Drupal\user\Entity\User::load($current_user->id())
);
// Merge cache tags
$tags = Cache::mergeTags($conf_one->getCacheTags(),$conf_two->getCacheTags());

Auto-placeholdering

To prevent highly dynamics content to invalided systematically page cache, Drupal replaces them with placeholders before building cache.
Each time the cache is requested their content will be rendered.

https://www.drupal.org/docs/8/api/render-api/auto-placeholdering

https://www.qed42.com/blog/lazy-builders-drupal8 Coeur du module BigPipe

Lazybuilder content could be:

  • defined manually with the #lazy_builder key:

    // Callback : class:method or (better) service:method
      return [
        '#lazy_builder' => ['hello_world.lazy_builder:renderSalutation', []],
        '#create_placeholder' => TRUE,
      ];
  • or be automatically detected by Drupal, following the rules defined into core/core.services.yml: 
    renderer.config:
    auto_placeholder_conditions:
      max-age: 0
      contexts: ['session', 'user']
      tags: []
    you could change them or add your own ones.

Using Drupal cache mechanisms in your custom code

Drupal already handles all these caches mecanisms, so "everything is cool"

But you will certainly needs to cache your own cusytom data, so here's cache API.

Static

  • Simple "caching" method using the fact that php static variables aren't destroyed at the end of a function execution.
  • drupal_static() could still be used for procedural code.
  • As for OO, don't forget that "services" are "singletons" and therefore their properties are persistents

Drupal cache Getter / Setter

function my_custom_data_processing();
  // Define my cache key
  $key = 'my_module' . ':' . __FUNCTION__ . ':' .
  \Drupal::languageManager()->getCurrentLanguage()->getId();
  // Does the cache exists ?
  if ($cache = \Drupal::cache()->get($key)) {
    $data = $cache->data;
  }
  // No cache: process the data and set the cache.
  else {
    $data = my_slow_data_process();
    \Drupal::cache()->set($key, $data);
  }
  return $data;
}

Duration

// Cache lifetime could be set (timestamp)
\Drupal::cache()->set($key, $data, REQUEST_TIME + 600);
On utilise les deux points (:) comme séparateurs hiérarchique.

Tags

// Merge tags
\Drupal::cache()->set(
  $key,
  $data,
  Cache::PERMANENT,
  [
    'tag1',
    'node:1',
    'config:system.menu',
    'config:my_module',
  ]
);
// Entities tags list:
$node->getCacheTags();
$entity_type->getListCacheTags(); //EntityTypeInterface
\Drupal\views\Entity\View::load('front')->getCacheTags();
;

Delete / Invalidate

// Deleting cache (fast)
\Drupal::cache()->delete('my_module:cached_data');
\Drupal::cache()->deleteMultiple([
  'my_module:key1',
  'my_module:key2',
  ...
]);
\Drupal::cache()->deleteAll();


// Tag invalidation  (my_tag counter is increase by one)
$cache_tag_invalidator->invalidateTags(['my_tag']);
Cache::invalidateTags(['my_tag']);

On cache writing, all tags invalidation counter are sum up to provide the cache checksum.

On cache reading, the checksum is compute again, if it differe, the cache is rebuild.

Chaque tag se voit attribuer un compteur qui est incrémenté à chacune de ses invalidations. Le checksum du contenu est simplement la somme de ces compteurs.

Multiples

\Drupal::cache()->getMultiple($keys);
\Drupal::cache()->setMultiple($items);

Return stale cache while building the new one

$cache = \Drupal::cache()->get('my-key', TRUE);
if ($cache && $cache->valid) {
  return $cache->data;
}
elseif (\Drupal::lock()->acquire('my-key')) {
  // Rebuild and set new data.
}
elseif ($cache) {
  // Someone else is rebuilding, work with stale data.
  return $cache->data;
}
else {
  // Wait or rebuild.
}

Cache Storages

Bins

Cache is split into containers:

  • default: Default, for small-ish, few key variations
  • data : Bigger caches, many key variations
  • discovery: Small, frequently used, usually for plugins and similar discovery processes
  • bootstrap: Drupal bootstraping.
  • render: HTML rendering cache
  • config: Used for caching configuration
  • static: Memory only, when persistence is not desired

Each could be linked to a specific storage

You could define you own bin as a service

yourmodule.services.yml:

  cache.your_bin:
    class: Drupal\Core\Cache\CacheBackendInterface
    tags:
      - { name: cache.bin }
    factory: cache_factory:get
    arguments: [your_bin]

and enable it in settings.php

// Use redis by default.
$settings['cache']['default'] = 'cache.backend.redis';

// Use the null backend to disable caching for certain bins.
$settings['cache']['bins']['render'] = 'cache.backend.null';
bootstrap est rarement invalidé.
Oui d'ailleurs c'est quoi un backend ?

Backends

Where cache are actually stored.

  • Core backends (core/core.services.yml):
    • Memory: not persisted across requests
    • Database: Default storage in the (SQL) database (invalidated, never deleted)
    • APCu: Shared-Memory inside the PHP process, not shared with drush/CLI/multiple servers
    • Null: for disabling cache (dev)
  • Contributed backends:
    • Slushi Cache: A database backend with a configurable max lifetime
    • Memcache : RAM object database server cache.backend.memcache_storage
    • Redis: RAM Key/value database server cache.backend.redis

ChainedFast Backend (cache.backend.chainedfast) allow to "chained" a fast backend with a slow one.
